# Authorized Help and Sticker Feedback Design

## Goal

Add two private-chat interactions without changing conversion behavior:

1. Authorized users can run /help to see available commands. The owner sees the
   same user help plus invitation administration commands.
2. After an authorized user sends a sticker, the bot acknowledges it before any
   download and continuously shows Telegram's typing action while processing.

## Authorization Boundary

/start remains the only command available before authorization because users
need it to redeem an invitation. Every other feature requires either an active
authorization record or OWNER_TELEGRAM_ID.

An unauthorized /help request returns only:

    请先使用邀请码启动 Bot。

It does not reveal the command list. Existing unauthorized sticker behavior is
unchanged and sends the same invitation-required response without acknowledging
or downloading the sticker.

## Help Content

A regular authorized user receives:

    可用指令：
    /help - 查看指令列表
    /start - 查看授权状态
    发送贴纸 - 自动转换为 PNG 或 GIF

The owner receives those lines plus:

    管理员指令：
    /invite - 创建邀请码
    /invites - 查看邀请码
    /revoke <邀请码> - 撤销邀请码

The text is generated by a pure help_text(is_owner: bool) function so content
and permission differences can be tested without Telegram network calls.

## Sticker Feedback Flow

For an authorized user or the owner, the sticker handler performs this exact
order:

1. Validate private chat, sender, sticker, and authorization.
2. Reply: 已收到贴纸，正在转换，请稍等。
3. Enter aiogram ChatActionSender.typing so the typing status is refreshed while
   work continues.
4. Download the source sticker, convert it, and upload the result document.
5. Leave the typing context after success or failure.
6. On failure, log the technical error and send the existing failure reply:
   转换失败，请稍后重试。

The acknowledgement is a separate permanent message. It is not edited or
deleted after completion.

## Files and Verification

app/handlers.py adds the help text helper, /help route, authorization helper, the
acknowledgement, and ChatActionSender.typing context.

tests/test_handlers.py verifies regular and owner help content and the shared
authorization decision. Handler behavior tests verify that unauthorized users
cannot reach the help list or acknowledgement, and that acknowledgement occurs
before Bot.get_file. The existing Python and Docker verification commands remain
unchanged.
